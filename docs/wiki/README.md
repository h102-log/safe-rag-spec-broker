## 1. 이 프로젝트는 무엇인가

### 한 줄 요약
**`domain-rag-eval-obs`** — 특정 폐쇄 문서집합(사내규정·매뉴얼·법령·기술문서)에 대해 LLM이
**근거를 갖고 답하고**, 그 답의 **품질을 숫자로 측정**하고, **모든 요청을 추적·계측**하는 RAG 시스템.

이름 그대로 세 조각이다.

- **domain-rag** — 도메인 문서 기반 RAG(검색증강생성)
- **eval** — 답의 품질을 정량 평가 (RAGAS)
- **obs** — 관측/추적 (Langfuse)

### 왜 그냥 "챗봇 데모"가 아닌가
RAG 데모 자체는 흔하다. 이 프로젝트의 차별점은 **"돌아가는 데모"가 아니라 "품질을 잴 수 있고
추적되는 프로덕션 RAG"**라는 점이다. 흔한 데모는 "검색이 한 번 작동함"만 보여주지만, 여기서는
신뢰성·비용·지연을 **지속적으로 측정**한다.

### 3개 레이어 구조

```
LAYER 1: RAG 파이프라인 (제품 본체)
  오프라인:  문서 → 파싱 → 청킹 → 임베딩 → 벡터DB
  온라인:    질문 → 검색 → 리랭킹 → 프롬프트 → 생성 → 인용
        │                              │
        │ 모든 단계를 trace로 감쌈       │ 결과를 채점
        ▼                              ▼
LAYER 3: 관측 (Langfuse)          LAYER 2: 평가 (RAGAS)
  요청별 trace, 토큰·비용·지연        검색축 vs 생성축 분리 진단
  프롬프트 버저닝, score 저장          골든셋 회귀 게이트 (CI)
```

- **Layer 1 (RAG 본체)**: 문서를 잘게 쪼개 벡터로 색인해두고(오프라인), 질문이 오면 관련
  조각을 검색해 LLM이 **출처를 붙여** 답한다.
- **Layer 2 (평가)**: 답이 맞는지를 숫자로. 특히 **"검색이 못 찾은 건지 / 생성이 지어낸
  건지"를 분리 진단**하는 게 핵심.
- **Layer 3 (관측)**: 블랙박스를 유리상자로. 어느 요청이 왜 실패했고 비용이 어디서 터지는지
  사후 추적한다.

### 핵심 원칙 (프로젝트 헌법 격, `CLAUDE.md`)
- 답변은 **반드시 검색된 문맥에 근거** — 파라메트릭 기억만으로 답 금지
- 답변에는 **인용(출처)을 반드시 포함**
- 모든 온라인 요청은 **Langfuse로 계측** (미설정 시 no-op이어도 데코레이터는 유지)
- **500페이지(20만 토큰) 미만이면 RAG를 만들지 마라** — 그냥 문서 전체를 프롬프트에 넣는 게
  더 싸고 정확

### 지금 어디까지 왔나 (`phase 0-rag-core`)

| 단계 | 파일 | 상태 |
|---|---|---|
| 파싱 | `app/ingest/parse.py` | ✅ (페이지 단위 인용 provenance까지) |
| 청킹 | `app/ingest/chunk.py` | ✅ |
| 임베딩 | `app/ingest/embed.py` | ✅ |
| 색인 | `app/ingest/index.py` | ✅ |
| 검색 | `app/retrieve/search.py` | ✅ (dense-only, 랭킹 검증까지) |
| 생성·인용 | `app/generate/` | ⏳ 아직 (미구현) |
| 평가 (RAGAS) | `eval/` | ⏳ 아직 |
| 관측 (Langfuse) | `observability/langfuse_setup.py` | 골격(no-op 스위치)만 |

즉 **"문서를 색인하고 검색해서 후보를 뽑는" 데까지 동작**하고, **LLM이 실제로 답을 생성·인용하는
부분과 평가·관측 레이어는 앞으로 붙일 차례**다.

## A. RAG란 무엇이고 왜 쓰는가

### RAG = Retrieval-Augmented Generation (검색증강생성)
LLM에게 질문을 던지기 **전에**, 관련 문서 조각을 먼저 **검색해서** 프롬프트에 **끼워넣고**,
LLM이 **그 근거만 보고 답하게** 하는 기법.

```
[그냥 LLM]  질문 ───────────────────────→ LLM ──→ 답 (모델이 "기억"으로 답함)
[RAG]       질문 → 관련 문서 검색 → 질문+문서 → LLM ──→ 답 (문서 근거로 답함)
```

### 왜 필요한가 — 그냥 LLM의 3가지 문제
1. **환각(hallucination)** — 모델은 모르는 걸 그럴듯하게 지어낸다. "우리 회사 연차 규정"을
   물으면 일반론을 창작한다.
2. **최신성** — 모델은 학습 시점 이후, 그리고 **회사 내부 문서**를 모른다.
3. **출처 없음** — "왜 그렇게 답했는지" 근거를 댈 수 없다. 규제 도메인에선 치명적.

RAG는 이 셋을 **"실제 문서를 찾아 붙이고, 그 문서만 근거로 삼게 함"**으로 해결한다. 그래서
이 프로젝트의 헌법이 **"답변은 반드시 검색된 문맥에 근거"** + **"인용을 반드시 포함"**이다.

### 언제 RAG를 쓰지 *말아야* 하는가 ⚠️
> **문서집합이 500페이지(≈20만 토큰) 미만이면 RAG를 만들지 마라.**

그 정도면 **문서 전체를 그냥 프롬프트에 통째로 넣는 게** 더 싸고 정확하다(프롬프트 캐싱 활용).
검색·임베딩·벡터DB는 그 경우 과잉 엔지니어링이다. RAG가 정당한 건 **"자주 바뀌는 데이터 +
출처 추적 필수 + 문서량이 많아 다 넣을 수 없음"** 세 축이 겹칠 때다.

## B. 전체 데이터 흐름 — 문서가 들어가서 답이 나오기까지

RAG는 시간대가 다른 **두 개의 국면**으로 나뉜다. 이 구분이 핵심이다.

### 🔵 국면 1: 오프라인 인덱싱 (미리, 딱 한 번)
문서를 검색 가능한 형태로 **미리 가공해 DB에 저장**해둔다. 사용자가 오기 전에 끝내는 작업.

```
문서파일(PDF/DOCX)
   │
   ① 파싱      parse.py    → 텍스트로 변환 (표·페이지 보존)
   │
   ② 청킹      chunk.py    → 512토큰짜리 조각들로 자름 (출처 꼬리표 유지)
   │
   ③ 임베딩    embed.py    → 각 조각을 1024차원 벡터로 (의미를 숫자로)
   │
   ④ 색인      index.py    → pgvector DB에 저장 (재실행해도 중복 안 됨)
   ▼
[Postgres + pgvector]  ← "검색 가능한 지식창고" 완성
```

### 🟢 국면 2: 온라인 질의 (사용자가 질문할 때마다)

```
질문 "연차는 며칠?"
   │
   ⑤ 질문도 임베딩 → 벡터로  (embed.py 재사용)
   │
   ⑥ 검색      search.py   → DB에서 "벡터가 가까운" 조각 top-5 찾음 (cosine 유사도)
   │
   ⑦ (예정) 생성·인용        → 찾은 조각을 LLM에 주고 출처 붙여 답 생성
   ▼
답 + 출처(source/page) + trace_id
```

**핵심 통찰**: ③ 문서 임베딩과 ⑤ 질문 임베딩은 **같은 모델**을 써야 한다. 문서와 질문을 같은
"의미 공간"에 놓아야 "가까움"을 비교할 수 있다. 그래서 `embed.py` 하나를 양쪽에서 재사용한다.

**현재 상태**: ①~⑥까지 구현 완료, ⑦(LLM 생성·인용)은 다음 페이즈. 그래서 지금 `/query`를
호출하면 **"답변"이 아니라 "검색된 조각들"**이 반환된다.

## C. 현재 코드 뜯어보기

레이어 순서: `app/ingest`(오프라인) → `app/retrieve`(온라인).

### ① `app/ingest/parse.py` — 문서 → 텍스트
```
parse_file(path) → list[ParsedDoc]
```
- **Docling**으로 PDF/DOCX를 파싱한다. **PyPDF 평문 추출 우회 금지** — 표와 레이아웃을
  markdown으로 보존하는 게 설계 근거(ADR). 표가 깨지면 그 뒤 모든 단계가 오염된다.
- **`ParsedDoc`에 `source`/`page`를 담는다**. 이게 나중에 인용(citation)의 씨앗.
- **페이지 단위로 쪼갠다** (`_to_parsed_docs`): PDF처럼 페이지가 있으면 페이지별로, DOCX처럼
  흐름 문서면 통째로 `page=None`. → 인용을 "몇 페이지"까지 찍기 위함.
- **지연 import** (`_converter`): docling은 torch/transformers를 끌고 오는 무거운 스택이라
  실제 파싱할 때만 로드한다. `@lru_cache`로 변환기를 재사용해 색인 루프에서 재로딩을 피한다.

### ② `app/ingest/chunk.py` — 텍스트 → 검색 단위 조각
```
chunk_docs(docs, chunk_size=512, overlap=64) → list[Chunk]
```
- **왜 자르나?** 문서 전체는 검색 단위로 너무 크다. 512토큰 조각으로 자르되 **64토큰 겹침
  (overlap)** 을 둬 의미가 경계에서 잘려나가는 걸 완화한다.
- **문자수가 아니라 토큰 기준** (`from_tiktoken_encoder`). README/ADR 결정.
- **출처 꼬리표 전파**: `source`/`page`를 각 `Chunk`로 이어붙인다. "인용 추적이 청킹에서
  끊기면 안 된다"는 CRITICAL 규칙의 구현.
- **`chunk_index`의 처리**: source별 연속 카운터라, 한 문서가 페이지별 여러 `ParsedDoc`로
  쪼개져 와도 인덱스가 겹치지 않는다. → DB의 `UNIQUE(source, chunk_index)` 멱등성을 지탱.

### ③ `app/ingest/embed.py` — 텍스트 → 벡터 (교체 가능 설계)
```
get_embedder() → Embedder  (기본 BGE-M3, dim=1024)
```
- **`Embedder` 프로토콜**: 소비 측(색인·검색)은 구체 클래스가 아니라 이 인터페이스에만 의존.
  → BGE-M3 ↔ OpenAI 모델 교체가 색인/검색 코드 변경 없이 가능(ADR-005).
- **정규화 안 함**: BGE-M3가 내부에서 이미 L2 정규화된 단위벡터를 주므로 cosine=내적.
  `ponytail:` 주석으로 "모델이 정규화를 멈추면 여기서 추가"라고 상한을 명시.
- 무거운 FlagEmbedding/torch는 **지연 import**.

### ④ `app/ingest/index.py` — 조각+벡터 → pgvector 저장
```
ensure_schema(conn);  index_chunks(chunks, embedder, conn) → 저장된 행 수
```
- **멱등성이 CRITICAL**: 같은 문서를 재색인해도 행이 중복되면 검색이 오염된다. →
  `UNIQUE(source, chunk_index)` + `ON CONFLICT DO UPDATE` **upsert**. 재실행하면 덮어쓴다.
- **HNSW 인덱스 + cosine**: 벡터 근접 검색(ANN)을 빠르게.
- **`conn`을 인자로 받음**: DB 드라이버(psycopg) import를 CLI로 격리 — 이 모듈 자체는 드라이버
  없이 import 가능.

### ⑤ `app/ingest/__main__.py` — 오프라인 파이프라인 오케스트레이터
```
python -m app.ingest --source ./docs
```
- ①~④를 **한 줄로 꿰는** CLI: 디렉토리를 순회하며 각 파일을 `parse→chunk→index`. 여기가
  무거운 의존성(psycopg, docling, torch)이 다 모이는 **유일한 지점**이다. 나머지 모듈이 가벼운 이유.

### ⑥ `app/retrieve/search.py` — 온라인 검색 (여기서 국면이 바뀜)
```
search(query, top_k=5, conn=None, embedder=None) → list[Retrieved]
```
- **질문을 임베딩해 pgvector cosine ANN**: `embedding <=> vector` 연산자로 가장 가까운 top_k
  조각을 뽑고, `1 - distance`를 유사도 점수로 반환.
- **`@observe(name="search")` 계측**: 온라인 호출이므로 Langfuse trace로 감싼다(CRITICAL).
  질문 임베딩 계측도 이 span에 포함.
- **현 페이즈는 dense-only** (ADR-002): BM25 하이브리드 + RRF + 리랭킹은 다음 페이즈. 지금은
  "검색된 청크만 반환, LLM 생성 없음".
- `Retrieved`에 `source`/`page`를 실어 인용 provenance를 온라인까지 이어준다.

### 곁다리: 계측·설정이 "없어도 안 죽는" 설계
- **`observability/langfuse_setup.py`**: Langfuse 키가 없으면 `@observe`가 **함수를 그냥
  통과시키는 no-op**. → 로컬/테스트가 Langfuse 없이도 돈다. 규칙은 "계측 데코레이터는 유지하되
  미설정 시 no-op".
- **`app/config.py`**: `.env`에서 설정 로드. LLM/관측 키는 **옵셔널** — 없어도 로컬 개발 가능.
- **`app/api.py`**: `/query`가 `search()`를 호출하고 결과+`trace_id`를 반환. 단, **아직 LLM
  답 합성은 없음** — 검색 결과를 그대로 돌려준다(⑦ 미구현).

### 관통하는 설계 원칙 3가지 (코드 전반에서 반복)
1. **인용 provenance를 절대 안 놓침** — `source`/`page`가 parse→chunk→index→search 전 구간을
   따라 흐른다.
2. **무거운 의존성은 지연 import + `conn` 주입** — 각 모듈이 torch/docling/psycopg 없이 import
   가능. 테스트가 가볍고 빠르다.
3. **멱등성 + 교체 가능성** — 재색인해도 안 깨지고(upsert), 임베딩 모델은 프로토콜로 갈아끼운다.
